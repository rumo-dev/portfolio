## 💡 技術的な拘り・実装のポイント

### 1. UniTaskによる軽量かつ制御しやすい非同期演出システム

従来のコルーチンや `Update` でのタイマー管理を廃止し、**UniTask** を全面的に導入。
敵のヒットストップ、カメラシェイク、時間差で発生する多段エフェクトなどをすべて非同期処理のタスクチェーンとして記述しました。
これにより、ソースコードが上から下に流れる一本の流れとして記述でき、演出の順番変更やタイミングの微調整（ミリ秒単位）を非常に容易にしました。

### 2. イベント駆動型のアクション・リアクションロジック

戦闘ロジックとエフェクト（音響・パーティクル・UI）の依存関係を疎結合にするため、**C#のイベント（Action/UnityEvent）**を活用したイベント駆動型システムを採用。
プレイヤーが攻撃した瞬間、攻撃ヒット時、死亡時などのステート変化時にのみイベントが発火し、各演出コンポーネントが自立的に動作する設計にしました。これにより、プランナーからの「ヒット時の音を変更してほしい」「ヒットエフェクトを追加してほしい」といった要望に対し、戦闘コードを一切変更せずにアタッチするだけで対応可能にしました。

### 3. GitHub Flowによる効率的なアセット統合

デザイナー・プランナーを含めた3人での円滑なコラボレーションを推進するため、**GitHub Flow** による運用を主導。
プログラムのブランチ運用だけでなく、デザイナーが作ったアセットやプレハブを安全に取り込むための「プレハブ分割ルール」を設定し、コンフリクト（競合）を極限まで低減。プロジェクト全体のビルドエラーをゼロに保ちながら開発を完了しました。

## 🔧 直面した課題と解決策

- **課題: 多数の敵が同時に攻撃やスキルを発動した際、エフェクト生成コストによるスパイク（カクつき）と、UniTaskのキャンセル処理漏れによるメモリリーク**
  エフェクトや弾丸を都度 `Instantiate` するとガベージコレクション（GC）が発生し、フレームレートが低下する課題が生じました。また、敵が死亡した際、実行中だった攻撃演出の UniTask がキャンセルされずに残り続け、存在しないオブジェクトを参照しようとするヌルリファレンスエラーが発生しました。

- **解決策:**
  1. オブジェクトを使い回す**オブジェクトプール（Object Pooling）**を自作。ゲーム開始時に一定数の弾丸と主要エフェクトを生成しておき、アクティブ/非アクティブを切り替えることで、戦闘中の動的生成コストをほぼゼロにし、60FPSの動作を維持。
  2. すべての UniTask 処理に `CancellationToken` を渡し、オブジェクトの破棄（`OnDestroy`）時にトークンをトリガーしてタスクを強制終了するライフサイクル管理を徹底。これにより、敵の死亡に伴うタスクのゾンビ化を完全に防ぎました。
